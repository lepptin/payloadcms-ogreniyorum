Payload 3, önceki sürümlerden farklı olarak, kendisini **Next.js uygulamasının bir parçası** olarak konumlandırmak için sıfırdan yeniden inşa edildi. Bu kararın ardında, modern web geliştirmenin iki temel dinamiğini tek bir çatı altında birleştirme hedefi yatıyor: **içerik yönetim paneli (admin UI)** ve **son kullanıcıya sunulan ön yüz (public frontend)**. Next.js’in App Router, React Server Components (RSC) ve Server Actions gibi yenilikleri, bu birleşimi yalnızca mümkün değil, aynı zamanda son derece performanslı ve geliştirici dostu hale getirdi.

## Kavramsal Açıklama: Neden Next.js?

Payload v2, Express tabanlı bağımsız bir sunucu olarak çalışıyordu. Admin paneli, istemci tarafında render edilen bir React uygulamasıydı (CRA benzeri) ve veri alışverişi REST/GraphQL API üzerinden sağlanıyordu. Bu mimari, ön yüzü Next.js ile geliştirilen bir projede **iki ayrı sunucu**, **iki ayrı derleme süreci** ve **iki ayrı veri katmanı** anlamına geliyordu. Payload’ın Local API’si bile, sunucu bağlamı gerektirdiği için Next.js’in getServerSideProps gibi ortamlarında kullanılabiliyordu ancak bu, yine de Payload sunucusunun ayrıca çalışmasını zorunlu kılıyordu.

Next.js 13.4 ile gelen App Router ise **tamamen yeni bir zihinsel model** sundu:
- **React Server Components (RSC):** Bileşenler sunucuda render edilir, istemciye HTML olarak gönderilir. Böylece veri çekme işlemleri doğrudan bileşen içinde, API katmanı olmadan yapılabilir.
- **Server Actions:** Form gönderimleri ve veri mutasyonları, ayrı bir API endpoint’i yazmadan, sunucuda çalışan fonksiyonlarla gerçekleştirilebilir.
- **Dosya tabanlı yönlendirme ve middleware:** Hem ön yüz hem de admin paneli aynı yönlendirme sistemini kullanabilir; yetkilendirme middleware ile merkezileştirilebilir.

Payload 3, işte bu modeli **kendi çekirdeğine** taşıdı. Artık Payload, Next.js projenize yüklenen bir pakettir ve `payload.config.ts` dosyasıyla yapılandırılır. Admin paneli, Next.js’in App Router’ı içinde `/admin` rotasında, tamamen RSC ile render edilen bir arayüz olarak sunulur. Bu sayede:
- Admin panelinin ilk yüklemesi için büyük JS bundle’larına gerek kalmaz; sunucu tarafında render edilmiş HTML doğrudan gelir.
- Veri okuma işlemleri (koleksiyon listeleme, doküman görüntüleme) RSC içinde Local API ile yapılır; ek bir HTTP isteği oluşmaz.
- Veri yazma işlemleri (doküman kaydetme, silme) Server Actions ile gerçekleştirilir; bu da otomatik olarak CSRF koruması ve sunucu tarafı validasyon sağlar.

## Mimari Açıklama: Payload 3 Nasıl Çalışır?

Payload 3’ün mimarisi, Next.js’in üç temel yapı taşıyla bütünleşir:

1. **Local API’nin Doğrudan Kullanımı (v2’den Kritik Fark):**  
   v2’de `payload.find()` gibi Local API fonksiyonları, Payload sunucusu başlatıldıktan sonra `req` nesnesiyle çağrılırdı. v3’te ise Payload, Next.js sunucusunun bir parçası olduğu için, **herhangi bir Server Component veya Server Action içinde, `req` olmadan** doğrudan `payload.find({ collection: 'posts' })` çağrısı yapabilirsiniz. Bu, Next.js geliştiricisinin veritabanı sorgusunu `page.tsx` içinde yapması kadar doğaldır.

2. **Veritabanı Adaptörleri (Yeni):**  
   Payload v2 yalnızca MongoDB’yi (Mongoose ile) desteklerken, v3 **Drizzle ORM** üzerinden PostgreSQL, SQLite, MySQL ve hâlâ MongoDB’yi (native sürücü ile) destekler. Bu, Next.js projelerinde yaygın kullanılan Vercel Postgres, Neon, PlanetScale gibi servislerle sorunsuz entegrasyon anlamına gelir. Veritabanı seçimi, `payload.config.ts` içinde bir adaptör tanımıyla yapılır.

3. **Kimlik Doğrulama ve Yetkilendirme (Yeni Yapı):**  
   v2’de auth, Express oturumları ve Passport.js stratejileriyle yönetilirdi. v3, Next.js’in **middleware** ve **server-side session** mekanizmalarını kullanır. Admin paneline erişim, `middleware.ts` içinde Payload’ın sağladığı `payload.auth` yardımcılarıyla kontrol edilebilir. Kullanıcı oturumu, Next.js’in `cookies()` fonksiyonu üzerinden okunur; bu da RSC ve Server Actions içinde tutarlı bir auth deneyimi sunar.

4. **Admin Panelin RSC ile Render Edilmesi:**  
   Payload’ın admin UI’ı, Next.js sayfaları gibi düşünülebilir. `/admin` rotası, Payload’ın iç bileşenlerini kullanan bir `page.tsx`’tir. Bu bileşenler RSC olduğu için, veri listeleri sunucuda oluşturulur, istemciye sadece HTML gönderilir. Etkileşimli kısımlar (ör. zengin metin editörü) ise `'use client'` sınırıyla istemci bileşenlerine ayrılır. Bu, Next.js’te kendi uygulamanızı geliştirirken uyguladığınız server/client bileşen ayrımının birebir aynısıdır.

## Next.js Geliştiricisi İçin Sezgisel Senaryolar

Next.js ile uygulama geliştirmiş bir yazılımcı, Payload 3’ü öğrenirken aşağıdaki tanıdık desenlerle karşılaşır. Bu senaryolar, “Payload’ı Next.js mimarisi mantığında” kavramayı sağlar.

### Senaryo 1: Blog Yazılarını Listeleyen Bir Sayfa

Next.js’te bir `app/blog/page.tsx` dosyası oluşturup, veritabanından yazıları çekmek için `db.query.posts.findMany()` gibi bir çağrı yaparsınız. Payload 3’te aynı işlem şöyle görünür:

```tsx
// app/blog/page.tsx (Server Component)
import { payload } from '@/payload' // Payload instance'ı

export default async function BlogPage() {
  const posts = await payload.find({
    collection: 'posts',
    limit: 10,
  })

  return (
    <ul>
      {posts.docs.map((post) => (
        <li key={post.id}>{post.title}</li>
      ))}
    </ul>
  )
}
```

Burada `payload.find()` doğrudan sunucuda çalışır; hiçbir API isteği yapılmaz. Next.js’in veri önbellekleme (cache) mekanizmaları da otomatik olarak devreye girer. Geliştirici, Payload’ı bir **ORM katmanı** gibi kullanır.

### Senaryo 2: Admin Panelinde Özel Bir Alan Ekleme

Next.js’te karmaşık bir form bileşeni yazarken, sunucu tarafında veri çekip istemci tarafında etkileşim eklemek için `'use client'` sınırını kullanırsınız. Payload’ın admin panelini özelleştirirken aynı prensip geçerlidir. Diyelim ki bir koleksiyona özel bir “konum seçici” alanı eklemek istiyorsunuz. Payload’ın sunduğu `Field` bileşenleri RSC’dir, ancak sizin interaktif harita bileşeniniz `'use client'` olmalıdır:

```tsx
// customComponents/MapField.tsx
'use client'
import { MapPicker } from 'some-map-library'

export function MapField({ value, onChange }) {
  return <MapPicker location={value} onChange={onChange} />
}
```

Bu bileşeni Payload konfigürasyonunda ilgili alana tanımlarsınız. Payload, sunucu tarafında formun statik kısmını render eder, istemci tarafında ise sadece bu interaktif bileşeni hydrate eder. Next.js’teki RSC modelinin birebir aynısıdır.

### Senaryo 3: İçerik Güncelleme İşlemi (Server Actions)

Next.js’te bir form gönderimini sunucuda işlemek için `app/actions.ts` içinde bir Server Action tanımlar, `action={updatePost}` şeklinde kullanırsınız. Payload 3’te admin paneli, doküman kaydetme işlemlerini zaten bu şekilde yapar. Kendi ön yüzünüzde bir “profil güncelleme” formu oluşturduğunuzda, aynı deseni Payload’ın Local API’si ile birleştirirsiniz:

```tsx
// app/actions.ts
'use server'
import { payload } from '@/payload'

export async function updateProfile(userId: string, data: any) {
  await payload.update({
    collection: 'users',
    id: userId,
    data,
  })
}
```

Bu action’ı `form action={updateProfile}` ile kullanırsınız. Payload’ın auth kontrolü, action içinde otomatik olarak çalışır; ayrıca bir API route yazmanız gerekmez.

### Senaryo 4: Yetkilendirme Kontrolü (Middleware)

Next.js projelerinde, belirli rotaları korumak için `middleware.ts` kullanılır. Payload 3’te admin panelinin kendisi de bu mekanizmayı kullanır. Örneğin, `/admin` rotasına sadece giriş yapmış editörlerin erişmesini istiyorsanız:

```ts
// middleware.ts
import { NextResponse } from 'next/server'
import { payload } from '@/payload'

export async function middleware(request) {
  const { user } = await payload.auth({ headers: request.headers })
  if (!user || user.role !== 'editor') {
    return NextResponse.redirect(new URL('/login', request.url))
  }
  return NextResponse.next()
}

export const config = { matcher: '/admin/:path*' }
```

Bu, Next.js’in yerleşik middleware yapısıdır; Payload sadece `payload.auth` ile oturum bilgisini sağlar. Geliştirici, kendi uygulamasındaki yetkilendirme mantığını hiç değiştirmeden Payload’a entegre eder.

## Sonuç

Payload 3’ün tamamen Next.js üzerine yeniden inşa edilmesi, onu bir “CMS eklentisi” olmaktan çıkarıp **Next.js uygulamasının doğal bir katmanı** haline getirdi. Bu dönüşüm sayesinde:
- Tek bir sunucu, tek bir derleme süreci, tek bir veri katmanı ile hem admin paneli hem de ön yüz yönetilir.
- React Server Components ve Server Actions, gereksiz API katmanlarını ortadan kaldırarak performansı artırır ve kod tekrarını azaltır.
- Next.js bilen bir geliştirici, Payload’ı öğrenirken yeni bir mimari paradigma öğrenmez; sadece bildiği Next.js desenlerini (server component’te veri çekme, Server Actions ile mutasyon, middleware ile auth) bir CMS bağlamında uygular.

Bu yaklaşım, özellikle içerik odaklı projelerde geliştirme hızını ve tutarlılığı radikal biçimde artıran bir devrim niteliğindedir.

Yeni bir Payload v3 projesi oluşturduğunuzda karşınıza çıkan dosya/klasör yapısı, Payload’un Next.js App Router’ı nasıl temel aldığını doğrudan yansıtır. Aşağıda tipik bir başlangıç projesinin hiyerarşisini ve her bileşenin rolünü bulacaksınız.

## 📁 Dosya & Klasör Hiyerarşisi (Treeview)

```
projem/
├── app/
│   ├── (payload)/                     # Payload'a özel route grubu
│   │   ├── admin/
│   │   │   ├── [[...segments]]/
│   │   │   │   ├── page.tsx           # Admin paneli yakalama sayfası
│   │   │   │   └── not-found.tsx      # Admin 404 sayfası
│   │   │   └── layout.tsx             # Admin paneli kök layout'u
│   │   ├── api/
│   │   │   ├── [...collection]/
│   │   │   │   └── route.ts           # REST API koleksiyon uç noktaları
│   │   │   ├── graphql/
│   │   │   │   └── route.ts           # GraphQL uç noktası
│   │   │   └── globals/
│   │   │       └── [...global]/
│   │   │           └── route.ts       # REST API global uç noktaları
│   │   └── layout.tsx                 # Payload route grubunun ortak layout'u
│   ├── (frontend)/                    # Kendi ön yüzünüz (isteğe bağlı)
│   │   ├── layout.tsx
│   │   └── page.tsx
│   └── layout.tsx                     # Uygulama kök layout'u
├── payload.config.ts                  # Payload yapılandırması (koleksiyonlar, globaller, eklentiler)
├── payload-types.ts                   # Otomatik üretilen TypeScript tipleri
├── next.config.js                    # Next.js yapılandırması (Payload eklentisi ile)
├── server.ts                         # Özel sunucu başlatma dosyası (isteğe bağlı, sıklıkla kullanılır)
├── package.json
└── tsconfig.json
```

> **Not:** `(payload)` ve `(frontend)` klasör adlarının başındaki parantezler, Next.js App Router’da **route group** (rota grubu) olduklarını belirtir. Bu klasörler URL yoluna etki etmez; yani `/admin` ve `/api` yolları doğrudan erişilebilir kalırken, `/frontend` gibi bir yol oluşmaz.

## Bileşenlerin Detaylı Açıklaması

### 1. `payload.config.ts` – Payload’un kalbi
Bu dosya, Payload’un tüm yapılandırmasını içerir:
- Veritabanı bağlantısı (v3’te MongoDB, Postgres, SQLite gibi adaptörler desteklenir)
- Koleksiyonlar (ör. `Posts`, `Users`)
- Globaller (ör. `Settings`)
- Eklentiler (örn. SEO, form builder)
- Admin paneli özelleştirmeleri, e‑posta ayarları, derinlik (depth) limitleri vb.

Payload v3, yapılandırmayı tamamen TypeScript ile yazmanıza olanak tanır ve bu dosya, hem sunucu hem de admin paneli tarafından kullanılır.

### 2. `server.ts` – Özel sunucu başlatma noktası
Bu dosya, Payload’ı Next.js ile birlikte programatik olarak başlatmak için kullanılır. Tipik içeriği:
- `payload.init()` ile Payload’u başlatır.
- Ardından Next.js’in `createServer()` fonksiyonunu çağırarak uygulamayı ayağa kaldırır.
- **Neden gerekli?** Payload v3, Next.js App Router üzerinde çalışsa da, Local API’yi (sunucu tarafında HTTP isteği olmadan Payload işlemleri yapma) kullanabilmek, veri tohumlama (seed) veya özel CLI komutları çalıştırmak için ayrı bir başlatma dosyasına ihtiyaç duyar. `server.ts` bu işlevleri yerine getirir. Geliştirme sırasında `next dev` doğrudan çalıştırılabilir; ancak üretim ortamında veya tohumlama işlemlerinde `server.ts` üzerinden başlatmak standarttır.

### 3. `app/(payload)/` – Payload’un Next.js App Router’a entegrasyonu
Bu route grubu, Payload’un tüm HTTP yüzeyini (admin UI + API) barındırır. Next.js’in **route group** özelliği sayesinde, Payload rotaları uygulamanızın geri kalanıyla çakışmaz; aynı kök segmenti (`/`) paylaşabilirsiniz.

#### a. `admin/[[...segments]]/page.tsx`
- **Opsiyonel catch‑all segment (`[[...segments]]`)** kullanır. Bu, `/admin` ve `/admin/collections/posts/123` gibi tüm alt yolların tek bir sayfa bileşeni tarafından yakalanmasını sağlar.
- Payload’un yönetim paneli, Next.js App Router’ın **React Server Components** altyapısı üzerine inşa edilmiştir. Bu sayfa, admin arayüzünü sunucu tarafında render eder ve istemciye HTML olarak gönderir; etkileşimli kısımlar ise istemci bileşenleri olarak hidrate edilir.
- `not-found.tsx` dosyası, admin paneli içinde geçersiz rotalar için özel 404 sayfası sunar.

#### b. `api/` – REST ve GraphQL uç noktaları
- `[...collection]/route.ts`: Dinamik koleksiyon rotalarını yönetir. Next.js **Route Handler** (`route.ts`) dosyaları ile `/api/posts`, `/api/posts/123` gibi istekler Payload’ın REST API’sine iletilir.
- `graphql/route.ts`: GraphQL uç noktasını (`/api/graphql`) aynı Route Handler mekanizmasıyla sunar.
- `globals/[...global]/route.ts`: Global belgelere (ör. site ayarları) REST erişimi sağlar.
- Bu yapı, Payload’ın v3 ile birlikte **Next.js API Routes**’u tamamen benimsediğini gösterir; artık ayrı bir Express sunucusu çalıştırmaya gerek yoktur.

#### c. `layout.tsx` (route grubu içinde)
- Payload’a ait tüm sayfalar (admin ve API) için ortak bir layout sağlar. Bu layout, Payload’un gerekli sağlayıcılarını (örn. `PayloadProvider`, yetkilendirme kontrolleri) enjekte eder ve admin panelinin temel HTML yapısını oluşturur.

### 4. `app/(frontend)/` – Kendi ön yüzünüz (opsiyonel)
Payload, başsız (headless) bir CMS olduğu için kendi ön yüzünüzü tamamen ayırabilirsiniz. `(frontend)` route grubu, `/` altında kendi sayfalarınızı, layout’larınızı ve bileşenlerinizi organize etmenizi sağlar. Payload’un rotalarıyla çakışmayı önlemek için route group kullanılması en iyi pratiktir.

### 5. `payload-types.ts` – Otomatik tip üretimi
Payload, koleksiyon ve global yapılandırmalarınıza dayanarak TypeScript tip tanımlarını bu dosyaya üretir. Bu sayede `payload.find()` gibi işlemlerde tam tip güvenliği elde edersiniz. v3’te bu üretim, geliştirme sunucusu çalışırken otomatik olarak güncellenir.

### 6. `next.config.js` – Next.js yapılandırması
Payload, Next.js’e bir **plugin** olarak entegre edilir. `next.config.js` içinde `withPayload` eklentisi kullanılır:
```js
const { withPayload } = require('@payloadcms/next-payload')
module.exports = withPayload({
  // Next.js konfigürasyonunuz
})
```
Bu eklenti, Payload’un admin paneli için gerekli webpack/derleme ayarlarını ve ortam değişkenlerini yönetir.

## Payload v3’ün Next.js App Router’ı Kullanma Biçimi – Özet

| Next.js Özelliği | Payload’daki Karşılığı |
|------------------|--------------------------|
| **Route Groups** `(folder)` | `(payload)` grubu, admin ve API rotalarını izole eder; URL’e etki etmez. |
| **Route Handlers** `route.ts` | REST ve GraphQL uç noktaları, Next.js API Routes üzerinden sunulur. |
| **Catch‑all Segments** `[[...segments]]` | Admin panelinin tüm alt sayfaları tek bir sayfa bileşeniyle yakalanır. |
| **React Server Components** | Admin paneli, sunucu tarafında render edilerek performans ve SEO avantajı sağlar. |
| **Custom Server** (`server.ts`) | Payload’un Local API’sini kullanmak, tohumlama yapmak veya özel başlatma mantığı eklemek için Next.js sunucusu programatik olarak başlatılır. |
| **TypeScript & Otomatik Tip Üretimi** | `payload-types.ts` ile tüm koleksiyon/global işlemlerinde uçtan uca tip güvenliği. |

Bu yapı, Payload v3’ün önceki sürümlerden en büyük farkını ortaya koyar: **Artık Express tabanlı bir sunucuya ihtiyaç duymaz; Next.js’in App Router mimarisine tamamen entegre olur.** Böylece hem geliştirme deneyimi sadeleşir hem de modern React ekosistemiyle kusursuz bir uyum yakalanır.

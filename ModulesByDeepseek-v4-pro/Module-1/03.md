PayloadCMS v3’te “Local API” kavramı, **yerel bir veritabanı değil**, sunucu tarafında doğrudan programatik olarak çağırabileceğiniz veri erişim katmanıdır. Payload’un REST ve GraphQL uç noktalarının altında yatan aynı iş mantığını, HTTP katmanını aradan çıkararak kullanmanızı sağlar. Aşağıda bu kavramı mimari bağlamıyla ve somut örneklerle açıklıyorum.

---

## 1. Kavramsal Açıklama: Local API Nedir, Ne Değildir?

- **Local API, Payload’un çekirdek operasyonlarının (CRUD, auth, dosya işlemleri vb.) doğrudan çağrılabilen programatik arayüzüdür.**  
  REST ve GraphQL API’leri, istemci ile sunucu arasında HTTP istek/yanıt döngüsü oluşturur. Local API ise aynı Node.js süreci içinde, Payload konfigürasyonunuzdan türetilmiş `payload` nesnesi üzerinden bu operasyonları çalıştırır.  
- **Bir “yerel veritabanı” değildir.** Veritabanı bağlantısı (PostgreSQL, MongoDB, SQLite vb.) Payload’un yapılandırmasında tanımlıdır ve Local API de tıpkı REST/GraphQL gibi aynı veritabanına erişir. Local API’yi kullanmak, veritabanını kopyalamaz ya da ayrı bir depolama katmanı oluşturmaz.  
- **Payload v3’te Local API, Next.js App Router ile derin entegrasyona sahiptir.** Sunucu bileşenlerinde (Server Components), Route Handler’larda, özel uç noktalarda veya harici sunucu script’lerinde doğrudan veri sorgulayıp manipüle edebilirsiniz. Admin paneli de artık bir Next.js uygulaması olduğu için, kendi içinde Local API’yi kullanır.

---

## 2. Mimari Açıklama: Local API, REST ve GraphQL Arasındaki İlişki

Payload v3’ün veri erişim mimarisi şu katmanlardan oluşur:

```
[ İstemci (Tarayıcı, Mobil, vb.) ]
        |
[ REST / GraphQL API ]   ← HTTP katmanı (serialization, authentication token, CORS vb.)
        |
[ Local API ]            ← Payload’un iç operasyon katmanı (doğrudan fonksiyon çağrıları)
        |
[ Veritabanı Adaptörü ]  ← PostgreSQL, MongoDB, SQLite (config’de belirtilen)
```

- REST ve GraphQL uç noktaları, gelen HTTP isteğini işler, erişim kontrolünü uygular ve ardından **Local API’deki karşılık gelen metodu çağırır** (örneğin `payload.find()`).  
- Local API, aynı erişim kontrolü (access control), hook’lar (beforeChange, afterRead vb.) ve doğrulama (validation) katmanlarından geçer. Yani Local API ile yaptığınız bir sorgu, REST ile yapılanla aynı güvenlik ve iş kurallarına tabidir.  
- Local API’yi doğrudan kullandığınızda HTTP serileştirme/deserializasyon, ağ gecikmesi ve JWT token yönetimi gibi ek yükler ortadan kalkar. Bu, özellikle sunucu tarafında zincirleme işlemler (örneğin bir webhook’ta birden fazla koleksiyonu güncellemek) veya Next.js Server Components ile veri çekme senaryolarında performans ve kod sadeliği sağlar.

## 3. Sezgisel Örnekler: Local API vs. REST API

Aşağıdaki örnekler, bir “posts” koleksiyonundan yayınlanmış yazıları çekmek için iki yaklaşımı karşılaştırır. Payload v3 konfigürasyonunuzun `payload.config.ts` ile tanımlandığını ve `posts` koleksiyonunun var olduğunu varsayıyorum.

### 3.1. Local API ile Sunucu Tarafında Doğrudan Erişim

**Next.js App Router’da bir Server Component içinde:**

```tsx
// app/posts/page.tsx (Server Component)
import { getPayload } from 'payload'
import config from '@payload-config' // Payload v3'in ürettiği config dosyası
import React from 'react'

export default async function PostsPage() {
  const payload = await getPayload({ config })

  // Local API: doğrudan find çağrısı
  const { docs: posts } = await payload.find({
    collection: 'posts',
    where: {
      _status: { equals: 'published' },
    },
    sort: '-createdAt',
  })

  return (
    <div>
      {posts.map((post) => (
        <article key={post.id}>
          <h2>{post.title}</h2>
          <p>{post.excerpt}</p>
        </article>
      ))}
    </div>
  )
}
```

**Özel bir Route Handler’da (API benzeri işlev):**

```ts
// app/api/custom/stats/route.ts
import { getPayload } from 'payload'
import config from '@payload-config'
import { NextResponse } from 'next/server'

export async function GET() {
  const payload = await getPayload({ config })

  // Local API ile birden fazla sorgu
  const [publishedPosts, draftPosts] = await Promise.all([
    payload.find({ collection: 'posts', where: { _status: { equals: 'published' } } }),
    payload.find({ collection: 'posts', where: { _status: { equals: 'draft' } } }),
  ])

  return NextResponse.json({
    publishedCount: publishedPosts.totalDocs,
    draftCount: draftPosts.totalDocs,
  })
}
```

**Ne kazandınız?**  
- Tek bir `getPayload` çağrısıyla Payload’un tüm yeteneklerine eriştiniz.  
- HTTP isteği yapmadınız; veritabanı sorguları doğrudan çalıştı.  
- Sunucu bileşeninde veriyi doğrudan render’a aktardınız; istemciye ek bir API çağrısı gerekmedi.

### 3.2. REST API ile İstemci Tarafından Erişim

Aynı veriyi REST üzerinden almak isterseniz, Payload otomatik olarak `/api/posts` uç noktasını sunar. İstemci tarafında (örneğin bir Client Component) şöyle bir çağrı yaparsınız:

```tsx
'use client'
import { useEffect, useState } from 'react'

export default function PostsPageClient() {
  const [posts, setPosts] = useState([])

  useEffect(() => {
    fetch('/api/posts?where[_status][equals]=published&sort=-createdAt')
      .then(res => res.json())
      .then(data => setPosts(data.docs))
  }, [])

  return (
    <div>
      {posts.map(post => (
        <article key={post.id}>
          <h2>{post.title}</h2>
          <p>{post.excerpt}</p>
        </article>
      ))}
    </div>
  )
}
```

**Farklar:**  
- İstemci bir HTTP isteği yapar → ağ gecikmesi oluşur.  
- Sorgu parametreleri URL’ye serileştirilir, yanıt JSON olarak deserialize edilir.  
- Yetkilendirme için genellikle bir token (cookie veya header) taşınır.  
- Sunucu tarafında REST endpoint’i yine Local API’yi çağırır; yani aynı iş mantığı çalışır, ancak araya HTTP katmanı girer.

### 3.3. Hibrit Kullanım: Local API ile Özel REST Uç Noktası Oluşturma

Local API’nin gücü, karmaşık iş akışlarını tek bir HTTP çağrısı arkasında birleştirmektir. Örneğin, bir kullanıcı kaydı sırasında hem `users` koleksiyonuna kayıt ekleyip hem de bir `profiles` dokümanı oluşturmak:

```ts
// app/api/register/route.ts
import { getPayload } from 'payload'
import config from '@payload-config'
import { NextResponse } from 'next/server'

export async function POST(request: Request) {
  const payload = await getPayload({ config })
  const body = await request.json()

  // Local API ile zincirleme işlem
  const user = await payload.create({
    collection: 'users',
    data: {
      email: body.email,
      password: body.password,
    },
  })

  const profile = await payload.create({
    collection: 'profiles',
    data: {
      owner: user.id,
      displayName: body.displayName,
    },
  })

  return NextResponse.json({ user, profile })
}
```

Burada istemci tek bir REST çağrısı yapar, ancak sunucu tarafında Local API sayesinde iki ayrı koleksiyon işlemi atomik olmasa da ardışık ve kontrollü biçimde yürütülür. Aynı işlemi sadece REST ile yapmaya kalksaydınız, istemcinin iki ayrı endpoint’e istek atması ve arada hata yönetimi yapması gerekirdi.

## 4. Önemli Notlar ve v3’e Özgü Detaylar

- **`getPayload` fonksiyonu**, Payload v3’te Local API’ye erişmenin standart yoludur. Sunucu ortamında (Next.js Route Handler, Server Component, harici Node.js script) `config` ile başlatılır. Payload’un admin paneli veya REST/GraphQL istekleri içinde ise `req.payload` üzerinden zaten hazır bir Local API örneği bulunur.  
- **Local API, veritabanı adaptöründen bağımsızdır.** İster PostgreSQL ister MongoDB kullanın, `payload.find()` aynı şekilde çalışır.  
- **Erişim kontrolü (access control) Local API’de de geçerlidir.** Eğer bir koleksiyonun `read` erişimi yalnızca giriş yapmış kullanıcılara açıksa, Local API ile yaptığınız sorguda da bir kullanıcı bağlamı (user) sağlamanız gerekir. `getPayload` ile başlatılan örnekte varsayılan olarak kimlik doğrulaması yoktur; bu nedenle sunucu bileşeninde herkese açık verileri çekerken sorun olmaz, ancak kullanıcıya özel işlemler için `req` nesnesini taklit etmeniz veya `payload.auth()` gibi yöntemleri kullanmanız gerekebilir.  
- **Local API, “yerel veritabanı” terimiyle karıştırılmamalıdır.** Payload v3, veritabanı olarak yalnızca config’de belirtilen sunucuyu kullanır. Local API, bu veritabanına erişim şeklinizi değiştirir, depolama katmanını değil.

## 5. Sonuç: Ne Zaman Hangi API?

| Kriter                      | Local API                                                                 | REST / GraphQL API                                     |
|-----------------------------|---------------------------------------------------------------------------|--------------------------------------------------------|
| **Konum**                   | Sunucu tarafı (Server Components, Route Handler, script)                 | İstemci tarafı (tarayıcı, mobil) veya harici servisler |
| **Performans**              | HTTP katmanı yok, daha hızlı                                             | Ağ gecikmesi ve serileştirme maliyeti var              |
| **Karmaşık iş akışları**    | Birden çok koleksiyon işlemini tek bir bağlamda birleştirmek için ideal  | İstemcinin birden çok istek atması gerekebilir         |
| **Güvenlik**                | Erişim kontrolü aynen uygulanır; bağlamı siz yönetirsiniz                | JWT/session token ile otomatik bağlam taşınır          |
| **Kullanım kolaylığı**      | Sunucu kodunda doğrudan fonksiyon çağrısı, tip güvenliği yüksek          | İstemcide `fetch` veya SDK ile standart HTTP           |

**Özetle:** PayloadCMS v3’te Local API, “sunucu tarafında yaşayan, HTTP olmadan doğrudan Payload operasyonlarını çağırabildiğiniz programatik katmandır”. REST/GraphQL API’lerinin temelini oluşturur ve size sunucu içi entegrasyonlarda benzersiz bir esneklik kazandırır. Yerel bir veritabanı değil, veritabanınıza açılan en kısa yoldur.

Payload CMS v3’te **Collections (Koleksiyonlar)**, veritabanındaki tablolara karşılık gelen, içerik türlerinizi tanımladığınız yapı taşlarıdır. Her koleksiyon, belirli bir veri modelini (örneğin blog yazıları, kullanıcılar, ürünler) temsil eder ve bu modelin alanlarını, doğrulama kurallarını, erişim kontrollerini ve yönetici panelindeki görünümünü tek bir yerde yapılandırmanıza olanak tanır.

Aşağıda sorularınızı sırasıyla, gerçek hayat örnekleriyle ve TypeScript’e yeni başlayanlar için kısa hatırlatmalarla cevaplıyorum.

---

## 1. Collections (Koleksiyonlar) Nedir?

Bir **Collection**, Payload’a “şu isimde bir içerik türüm var, alanları şunlar, bu alanlar şu kurallara uysun, admin panelde şöyle görünsün” dediğiniz yapılandırma nesnesidir. Payload, bu tanıma göre veritabanında tablo oluşturur, otomatik CRUD API’leri (REST ve GraphQL) üretir ve admin panelini şekillendirir.

**Gerçek hayat örneği:** Bir blog yazısı koleksiyonu.

```typescript
// TypeScript hatırlatması: `export const` ile sabit bir değişken tanımlıyoruz.
// `CollectionConfig` tipi, Payload'ın koleksiyon yapılandırması için beklediği şekli belirtir.
import { CollectionConfig } from 'payload/types';

export const Posts: CollectionConfig = {
  slug: 'posts', // koleksiyonun benzersiz adı (API uç noktası ve veritabanı tablosu için)
  fields: [
    {
      name: 'title',
      type: 'text',
      required: true,
    },
    {
      name: 'content',
      type: 'richText',
    },
  ],
};
```

Bu tanım, `posts` adında bir koleksiyon oluşturur; `title` (zorunlu metin) ve `content` (zengin metin) alanlarına sahiptir. Payload otomatik olarak `id`, `createdAt`, `updatedAt` gibi sistem alanlarını ekler.

## 2. Koleksiyon Alan Tipleri

Payload v3, içeriğinizi şekillendirmek için zengin bir alan tipi seti sunar. Her alan tipi, kendine özgü seçeneklerle birlikte gelir. En sık kullanılanları gerçek hayat senaryolarıyla inceleyelim.

### `text` – Metin
Kısa metinler (başlık, isim, slug vb.) için.

```typescript
{
  name: 'title',
  type: 'text',
  required: true,
  minLength: 5,
  maxLength: 100,
}
```

### `number` – Sayı
Fiyat, stok adedi, sıralama gibi sayısal değerler.

```typescript
{
  name: 'price',
  type: 'number',
  required: true,
  min: 0,
}
```

### `date` – Tarih
Yayın tarihi, etkinlik zamanı gibi tarih/saat alanları.

```typescript
{
  name: 'publishedAt',
  type: 'date',
  admin: {
    date: {
      pickerAppearance: 'dayAndTime', // gün ve saat seçimi
    },
  },
}
```

### `richText` – Zengin Metin
Biçimlendirilmiş içerik (kalın, italik, bağlantı, resim ekleme vb.) için. Payload, zengin metin editörü olarak Lexical’i kullanır (v3’te Slate’ten Lexical’e geçiş yapılmıştır).

```typescript
{
  name: 'body',
  type: 'richText',
  required: true,
}
```

### `upload` – Dosya Yükleme
Resim, PDF, video gibi medya dosyalarını yönetmek için. Bu alan tipi, koleksiyonun kendisini bir “medya koleksiyonu” yapar. Genellikle ayrı bir `media` koleksiyonu tanımlanır ve diğer koleksiyonlardan `upload` alanı ile ilişkilendirilir.

```typescript
// Medya koleksiyonu
export const Media: CollectionConfig = {
  slug: 'media',
  upload: {
    staticDir: 'media', // dosyaların sunucuda saklanacağı klasör
    mimeTypes: ['image/*'], // sadece resim dosyaları
  },
  fields: [],
};

// Başka bir koleksiyonda kullanımı
{
  name: 'featuredImage',
  type: 'upload',
  relationTo: 'media', // hangi upload koleksiyonuna bağlanacağı
  required: true,
}
```

### `relationship` – İlişki
Koleksiyonlar arası bağlantılar kurar. Örneğin bir blog yazısının kategorisi veya yazarı.

```typescript
{
  name: 'category',
  type: 'relationship',
  relationTo: 'categories', // bağlanacak koleksiyon slug'ı
  hasMany: false, // bire-bir ilişki (true olursa birden çok kategori)
}
```

### `array` – Dizi
Tekrarlayan alt alan grupları (örneğin ürün özellikleri, SSS maddeleri). Her dizi elemanı kendi içinde birden çok alan barındırabilir.

```typescript
{
  name: 'faqItems',
  type: 'array',
  fields: [
    {
      name: 'question',
      type: 'text',
      required: true,
    },
    {
      name: 'answer',
      type: 'richText',
    },
  ],
}
```

### `blocks` – Bloklar
Esnek içerik yapıları için. Kullanıcı, önceden tanımlanmış blok şablonlarından seçerek sayfa içeriğini oluşturur (örneğin: metin bloğu, resim galerisi, alıntı, CTA butonu). Her blok kendi alan setine sahiptir.

```typescript
{
  name: 'layout',
  type: 'blocks',
  blocks: [
    {
      slug: 'textBlock',
      fields: [
        { name: 'heading', type: 'text' },
        { name: 'content', type: 'richText' },
      ],
    },
    {
      slug: 'imageBlock',
      fields: [
        { name: 'image', type: 'upload', relationTo: 'media' },
        { name: 'caption', type: 'text' },
      ],
    },
  ],
}
```

### `json` – JSON
Yapılandırılmış ancak şeması sabit olmayan veriler için. Örneğin üçüncü parti bir servisten gelen ham yanıtı saklamak.

```typescript
{
  name: 'externalMetadata',
  type: 'json',
}
```

### Diğer alan tipleri
- `email`, `password` (auth koleksiyonlarında kullanılır)
- `checkbox` (boolean)
- `select`, `radio` (seçenek listeleri)
- `point` (coğrafi koordinat)
- `code` (kod blokları)
- `row` / `collapsible` / `tabs` (alan gruplama, görsel düzenleme)

## 3. Koleksiyon Alan Doğrulama (Validation)

Doğrulama, alan düzeyinde veya koleksiyon düzeyinde yapılabilir. Alan düzeyinde `validate` fonksiyonu ile özel kurallar tanımlanır. Fonksiyon `true` dönerse geçerli, bir string dönerse hata mesajı olarak gösterilir.

**Örnek: Başlık uzunluğu ve özel format kontrolü**

```typescript
{
  name: 'title',
  type: 'text',
  required: true,
  validate: (value) => {
    // TypeScript: value parametresi string | undefined olabilir.
    if (!value) return 'Başlık zorunludur.';
    if (value.length < 10) return 'Başlık en az 10 karakter olmalı.';
    if (!/^[A-Z]/.test(value)) return 'Başlık büyük harfle başlamalı.';
    return true; // geçerli
  },
}
```

**Koleksiyon düzeyinde doğrulama:** Birden çok alanı birlikte kontrol etmek için `validate` fonksiyonu koleksiyon yapılandırmasının en üstüne yazılır. Bu fonksiyon, tüm veriyi (`data`) ve orijinal veriyi (`originalDoc`) parametre olarak alır; hata durumunda string döndürür.

```typescript
export const Products: CollectionConfig = {
  slug: 'products',
  fields: [
    { name: 'price', type: 'number' },
    { name: 'discountPrice', type: 'number' },
  ],
  validate: (data) => {
    if (data.discountPrice && data.discountPrice >= data.price) {
      return 'İndirimli fiyat, normal fiyattan düşük olmalıdır.';
    }
    return true;
  },
};
```

## 4. Koleksiyonlarda Hooks (Kanca) Nedir ve Nasıl Kullanılır?

Hooks, bir dokümanın yaşam döngüsünde belirli noktalarda çalışan işlevlerdir. Payload v3’te koleksiyon düzeyinde şu hook’lar bulunur:

- `beforeChange` – doküman oluşturulmadan/güncellenmeden önce
- `afterChange` – doküman başarıyla oluşturulduktan/güncellendikten sonra
- `beforeRead` – doküman okunmadan önce
- `afterRead` – doküman okunduktan sonra
- `beforeDelete` – silme işleminden önce
- `afterDelete` – silme işleminden sonra
- `beforeLogin` / `afterLogin` (auth koleksiyonlarında)
- `beforeValidate` – doğrulama öncesi

Hook fonksiyonları, `args` nesnesi alır (`data`, `req`, `operation` vb.) ve değiştirilmiş veriyi dönebilir.

**Gerçek hayat örneği: Otomatik slug oluşturma (`beforeChange`)**

```typescript
export const Posts: CollectionConfig = {
  slug: 'posts',
  fields: [
    { name: 'title', type: 'text', required: true },
    { name: 'slug', type: 'text', unique: true, admin: { readOnly: true } },
  ],
  hooks: {
    beforeChange: [
      ({ data, operation }) => {
        // TypeScript: data nesnesi, operation 'create' | 'update'
        if (operation === 'create' && data.title) {
          data.slug = data.title
            .toLowerCase()
            .replace(/ /g, '-')
            .replace(/[^\w-]+/g, '');
        }
        return data; // değiştirilmiş veriyi döndür
      },
    ],
  },
};
```

**Örnek: `afterChange` ile loglama veya dış servis bildirimi**

```typescript
hooks: {
  afterChange: [
    ({ doc, operation }) => {
      console.log(`${operation} işlemi yapıldı: ${doc.id}`);
      // Burada e-posta gönderme, arama indeksi güncelleme gibi işlemler yapılabilir.
    },
  ],
}
```

**Örnek: `beforeRead` ile hassas veriyi gizleme**

```typescript
hooks: {
  beforeRead: [
    ({ doc, req }) => {
      // Eğer kullanıcı admin değilse, 'internalNotes' alanını kaldır
      if (!req.user || req.user.role !== 'admin') {
        delete doc.internalNotes;
      }
      return doc;
    },
  ],
}
```

## 5. Koleksiyonlarda Access Kontrolü (CRUD)

Access kontrol, koleksiyon üzerindeki **C**reate (oluşturma), **R**ead (okuma), **U**pdate (güncelleme) ve **D**elete (silme) işlemlerini kimlerin yapabileceğini belirler. Payload v3’te her koleksiyon için `access` nesnesi tanımlanır. Her bir işlem için bir fonksiyon yazılır; fonksiyon `true` dönerse izin verilir, `false` dönerse engellenir. Fonksiyon `req` (istek nesnesi) parametresini alır, böylece oturum açmış kullanıcıya (`req.user`) göre karar verilebilir.

**Gerçek hayat örneği: Sadece giriş yapmış kullanıcılar yazı oluşturabilir, herkes okuyabilir, sadece yazar veya admin güncelleyip silebilir.**

```typescript
export const Posts: CollectionConfig = {
  slug: 'posts',
  access: {
    create: ({ req }) => {
      // TypeScript: req.user, oturum açmış kullanıcıyı temsil eder (varsa).
      return Boolean(req.user); // true dönerse izin ver
    },
    read: () => true, // herkese açık
    update: ({ req, id }) => {
      if (!req.user) return false;
      if (req.user.role === 'admin') return true;
      // Sadece kendi dokümanını güncelleyebilir (dokümanın 'author' alanı varsa)
      // Bu kontrolü hook veya alan seviyesinde yapmak daha doğru olabilir; burada basit örnek.
      return { author: { equals: req.user.id } }; // query constraint
    },
    delete: ({ req }) => {
      return req.user && req.user.role === 'admin';
    },
  },
  fields: [
    { name: 'title', type: 'text' },
    { name: 'author', type: 'relationship', relationTo: 'users' },
  ],
};
```

**Not:** `update` ve `delete` erişim fonksiyonları, bir sorgu kısıtlaması (örneğin `{ author: { equals: req.user.id } }`) dönebilir. Bu, işlemi tamamen engellemek yerine yalnızca belirli koşulları sağlayan dokümanlarla sınırlar.

**Field-level access (alan düzeyinde erişim):** Her alan için `access` özelliği ile okuma/güncelleme izinleri ayrı ayrı kontrol edilebilir.

```typescript
{
  name: 'internalNotes',
  type: 'text',
  access: {
    read: ({ req }) => req.user?.role === 'admin',
    update: ({ req }) => req.user?.role === 'admin',
  },
}
```

## 6. Admin Panel Özelleştirmeleri (Grup, Açıklama, Liste Görünümü)

Payload’un yönetici paneli, koleksiyon ve alan tanımlarınıza göre otomatik oluşturulur. `admin` özelliği ile görünüm ve davranışı özelleştirebilirsiniz.

### Alanları Gruplama (`group`)

Alanları mantıksal gruplara ayırmak için `type: 'group'` kullanılır. Grup içine `fields` dizisi yerleştirilir. Admin panelde sekmeli (tabs) veya gruplanmış olarak görünür.

```typescript
{
  name: 'seo',
  type: 'group',
  admin: {
    description: 'Arama motoru optimizasyonu ayarları',
  },
  fields: [
    { name: 'metaTitle', type: 'text' },
    { name: 'metaDescription', type: 'textarea' },
  ],
}
```

### Sekmeler (`tabs`)

Daha büyük formları sekmelere bölmek için `type: 'tabs'` kullanılır. Her sekme bir `Tab` nesnesidir.

```typescript
{
  type: 'tabs',
  tabs: [
    {
      label: 'İçerik',
      fields: [
        { name: 'title', type: 'text' },
        { name: 'body', type: 'richText' },
      ],
    },
    {
      label: 'SEO',
      fields: [
        { name: 'metaTitle', type: 'text' },
        { name: 'metaDescription', type: 'textarea' },
      ],
    },
  ],
}
```

### Açıklama (`description`)

Her alan veya koleksiyon için `admin.description` ile yardımcı metin eklenebilir. Kullanıcıya alanın ne işe yaradığını açıklar.

```typescript
{
  name: 'slug',
  type: 'text',
  admin: {
    description: 'URL’de görünecek benzersiz tanımlayıcı. Boş bırakılırsa başlıktan otomatik üretilir.',
  },
}
```

### Liste Görünümü (`listView`)

Admin paneldeki liste tablosunda hangi sütunların gösterileceğini `admin.components.listView` veya daha basit olarak `admin.defaultColumns` ile belirleyebilirsiniz. Payload v3’te `admin` altında `useAsTitle` (başlık olarak kullanılacak alan) ve `defaultColumns` yaygındır.

```typescript
export const Posts: CollectionConfig = {
  slug: 'posts',
  admin: {
    useAsTitle: 'title', // listede ve ilişkilerde gösterilecek ana alan
    defaultColumns: ['title', 'author', 'publishedAt', 'status'],
    description: 'Blog yazılarınızı yönetin',
  },
  fields: [ /* ... */ ],
};
```

**Gelişmiş liste özelleştirmesi:** `admin.components` ile özel React bileşenleri enjekte edilebilir, ancak bu ileri düzey bir konudur. Temel ihtiyaçlar için `defaultColumns` ve `useAsTitle` yeterlidir.

### Diğer admin özelleştirmeleri

- `admin.hidden`: Koleksiyonu veya alanı admin panelden tamamen gizler.
- `admin.readOnly`: Alanın sadece okunabilir olmasını sağlar.
- `admin.position`: Alanların sıralamasını belirler (sidebar, main).
- `admin.condition`: Belirli bir koşula göre alanı göster/gizle (örneğin başka bir alanın değerine bağlı).

```typescript
{
  name: 'discountPrice',
  type: 'number',
  admin: {
    condition: (data, siblingData) => siblingData.hasDiscount === true,
    description: 'Sadece "İndirim Var" işaretlendiğinde görünür.',
  },
}
```

## TypeScript Hatırlatmaları (Yeni Başlayanlar İçin)

- **Tip belirtme:** `const variable: Type = value;` şeklinde değişkene tip eklenir. Payload’da `CollectionConfig` tipini içe aktararak koleksiyon nesnesinin şeklini garanti altına alırız.
- **Fonksiyon parametre tipleri:** `(param: string)` gibi. Hook ve access fonksiyonlarında `({ data, req })` şeklinde destructuring kullanılır; tipler Payload tarafından sağlanır, siz yazmasanız da olur.
- **Union types:** `string | undefined` bir değer string olabilir veya tanımsız olabilir. `if (!value)` kontrolü ile undefined durumunu ele alırız.
- **Optional chaining:** `req.user?.role` ifadesi, `req.user` varsa `.role` değerini alır, yoksa `undefined` döner; hata fırlatmaz.
